Step 1 — 设计说明

核心
两个按钮，两条路径，互不干扰
「管理订阅」→ 直接跳系统设置（变更方案、查看详情、无挽留）
「取消订阅」→ 进入挽留流程（确认页 → 问卷 → 跳转）
UX
「管理订阅」蓝色卡片，视觉优先级更高
大部分用户点管理只是想看到期时间或换方案，不应被拦截
UX
「取消订阅」灰色卡片，弱化但不隐藏
合规要求取消入口可见，但视觉上不引导用户往这走
文案
价格 + 扣费日 + 使用天数，透明展示
价格透明降低「怕扣钱」焦虑；128 天制造沉没成本
核心
先告诉用户「你会失去什么」，再问原因
用具体权益清单触发损失厌恶，比直接弹问卷更有效
文案
「确定要离开吗？」+ 具体到期日期
情感化但不过度煽情；日期具体化让用户知道还剩多少时间
UX
权益清单最后一条放使用天数
前三条是功能损失，最后一条是情感损失（128 天白费了）
UX
V1 可选：可跳过此页直接到问卷
追求最小 MVP 可砍，但建议保留——开发成本极低，挽留增量约 5-8%
核心
V1 核心数据采集页 — 只收集，不做分支挽留
2-4 周数据回收后再决定哪个原因值得做针对性挽留
UX
不强制选择，未选也能点「继续取消」
reason_code 为空本身也是信号（用户对问卷不耐烦）
UX
没有返回按钮，只有两个出口
保留 → Step 5 · 继续取消 → Step 4 跳转系统设置
文案
6 个选项，选「其他」展开输入框（选填）
输入框不强制填写，给愿意表达的用户一个出口
核心
两类渠道，两种取消体验
商店渠道（iOS/华为/鸿蒙/小米）可调用 API 直接跳转；支付宝/微信渠道（OPPO/VIVO）无法直接跳转，需引导用户手动操作
UX
商店渠道：调用平台 API 跳转订阅管理页
iOS → StoreKit 2 · 华为 → 华为支付 · 鸿蒙 → HMS IAP · 小米 → 小米支付
UX
支付宝/微信渠道：展示图文取消教程
OPPO/VIVO 用户需要去支付宝或微信的「自动扣款」中取消，App 内展示分步图文引导
UX
跳转失败 → 统一降级为图文教程
任何渠道跳转异常时都有兜底方案，不让用户卡在死胡同
UX
保留成功 → 正向反馈闭环
不要默默关页面，给用户「做了正确选择」的温暖感
文案
再次展示 128 天，强化留下的理由
让用户带着正面情绪回到 App，而非「差点被套路」的不适
